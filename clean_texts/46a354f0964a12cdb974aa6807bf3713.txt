                        Switching to SPICE display made my Proxmox VMs a lot more responsive
               

            I don’t need the SPICE client just to access their GUIs, either
    
During my early Proxmox days, I was always trying new things to enhance the performance of my virtual machines – partly because I wanted to create dev environments for my coding tasks, and also because the thought of turning virtualized Linux setups as responsive as my everyday system seemed equal parts impressive and unhinged. Allocating plenty of system resources and tweaking the CPU settings definitely improved the experience, but something was still amiss.
✕
 Remove Ads
Moving the cursor was somewhat sluggish, no matter what OS I used for the VM. Likewise, launching anything on virtual machines with desktop environments would take slightly longer than it should. And these problems didn’t affect a specific system, either. Switching from my old PC to my Xeon server – or even arming my current gaming PC with Proxmox via a spare SSD – couldn’t fix the lag issues. Heck, even enabling the QEMU guest tools and passing a graphics card (one at a time, of course) to my virtual machines failed to make my everyday tasks feel more responsive.
That’s when I came across some forum posts claiming that the default Display option might be the culprit. And well, changing it to SPICE instantly made my virtual machines snappier. The cursor no longer left a trail when swiping across the screen, while switching between apps became a lot more seamless.

Here’s the fun part: if you try clicking on the Console button in the top right corner of the Proxmox web UI, it will download a .vv file on your client, and you’ll need to use it with a remote access app (something like Virt-viewer) to access the virtual machine’s GUI elements. However, you can sidestep this process entirely by using the Console tab on the left side of the screen, right where you’d find typical VM settings. While you probably won’t be able to use advanced SPICE features with this workaround, it’s my go-to method for tinkering with disposable virtual machines when I don’t want to configure access setups on my client devices.
✕
 Remove Ads

            It also fixed a weird issue with high-resolution desktop environments
    
Enhancing the snappiness of my virtual machines wasn’t the only advantage of enabling SPICE. When I used to change the display resolution on my Linux VMs (specifically instances relying on Debian-based distros) past 1440p, I’d start to encounter weird bugs. If the virtual machine crashed out of nowhere, I’d end up with a black screen after the initial login page – and this would only happen when I’d bump the resolution up a notch. Heck, I’d have VMs with this display issue even after a clean shutdown, and the only way to resolve it involved reloading to an old snapshot with the default 1024x576 resolution.
After I started switching my virtual machines to SPICE, this problem never reared its ugly head. It’s such a minor fix, but it solved a massive problem that troubled me countless times during my early Proxmox days.

                        But you might want to use different tools for remote gaming VMs
               

            Parsec for a simple setup, Sunshine + Moonlight if you don’t mind tinkering a bit
    
I tend to switch the Display setting to SPICE on almost all Linux virtual machines, but my Windows 11 remote gaming and dev VMs are an exception. I opt for Windows 11 Pro for virtual machines, and it’s mostly so I can access them using RDP. It works exceedingly well for everyday tasks, even when I use the Windows App on my MacBook.
