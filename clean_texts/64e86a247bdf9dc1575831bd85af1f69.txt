For old-timers like yours truly, who’ve created (and disassembled) multiple Proxmox rigs, manually removing the enterprise repo and adding its community equivalent is just another post-installation task.

                        Leaving the display setting at its default value renders your VMs less responsive
               

            I use the SPICE option for most virtual machines
    
Let’s say you’re trying to build a dev VM that’s armed with all the coding tools you need for everyday tasks. You’ve probably tossed a bunch of processor cores and system memory at it to avoid running into performance issues while working on your projects. Unfortunately, if you’re using the basic console on the web UI to access the virtual machine, you’re bound to encounter a slightly lagging mouse. Attempting to change the resolution past the 1080p mark might result in screen glitches. This might just be me, but I’ve also had VMs with higher display resolutions refuse to launch the desktop environment after restarting them if I left them at the default settings.
✕
 Remove Ads
Well, the fix for every single problem lies in the display option. You’ll want to modify the display setting in the Hardware tab (or graphics card, as it’s called in the VM creation wizard) of your virtual machine to something other than Default. I’ve had the most luck with SPICE, as it not only makes the cursor more responsive, but also lets me select past the 1440p resolution without the desktop environment bug. Better yet, you don’t even need to configure the SPICE app to use this feature; simply use the console tab under the Summary section instead of the button at the top of the Proxmox page the next time you need to access your virtual machine from PVE’s web UI.

                        The host CPU type is ideal for single-node setups
               

            Unless you have compatibility issues, you might want to ditch x86-64-v2-AES
    
If you’re looking for another obscure tweak that can improve your performance, you might want to modify the CPU type option for your virtual guest. By default, Proxmox assigns the x86-64-v2-AES CPU to your virtual machines, and truth be told, it has some perks. Since it’s a generic processor, it provides a stable experience in practically every OS you might want to run inside a VM. But the real advantage of x86-64-v2-AES is that migrating your virtual machine across cluster nodes with entirely different specs won’t result in compatibility issues if you have this CPU type equipped.
✕
 Remove Ads
But for folks who want the maximum performance in their modern Linux or Windows virtual machines, switching to the host CPU might be the better choice. With x86-64-v2-AES, you’re passing your CPU through an emulation layer, while the host type directly exposes the underlying architecture to your VM. Plus, you also get to use the full range of extensions and instruction sets supported by the host processor on your dev virtual machine. Tweaking this setting is just as simple, as you can find it within the Processor section of your virtual machine’s Hardware tab.

                        Tweaking the CPU governor profile can lower your energy bills
               

            The slight performance trade-off in burst operations is worth it
    
This may sound counterintuitive after all the responsiveness-enhancing tweaks I’ve mentioned so far, but it’s a good idea to switch the CPU governor profile from performance to powersave. If you haven’t heard of it, a CPU governor profile is responsible for controlling the processor’s speeds at specific workloads. Proxmox sets performance as the default CPU governor, which provides a snappy experience in burst tasks at the cost of increasing your server’s power consumption even while it’s not active.

Meanwhile, the powersave profile throttles down your CPU speeds during the idle state, allowing you to cut down on the energy siphoned by your server. The best part? The performance impact of switching the CPU governor is pretty minimal for intensive tasks, as your processor can still hit high frequencies. It’s just that you’ll experience a slight delay (of a few seconds, at most) during burst tasks, where your server runs for a couple of minutes, only to go back to sleep once the workload is complete. But if you’re on an old system, especially one with enterprise-grade hardware, this setting can be a game-changer for reducing your passive energy consumption. You can either rely on the 
PVE CPU Scaling Governor script
 from the Proxmox VE Helper Scripts repo or execute
 echo powersave | tee /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor
 within the Shell tab of your server to switch to the powersave profile.

            Don’t forget about (trustworthy) third-party repos, too
    
While we’re on this subject, Proxmox has one of the most talented communities out of every other virtualization platform. So much so that you’ll find a plethora of cool repositories by developers for your Proxmox environment. Now, I wouldn’t recommend blindly installing every script you see on GitHub. But 
PECU
, 
Proxmox VE Helper
, and other trustworthy sources have some neat templates and scripts that can make your Proxmox journey a lot easier. And many of these LXC templates are simpler to deploy than the TurnKey ones available on a barebones Proxmox setup.
