Despite spending most of my time buried under my 
Proxmox
 nodes and TrueNAS storage rig, I love tinkering with other virtualization platforms, container runtimes, and NAS-centric distros in my spare time. But since I’ve only ever used Linux and Windows (technically Mac, if you include Apple Containers) platforms, you can imagine my surprise when I heard of a home lab platform designed on top of FreeBSD.
✕
 Remove Ads
Now, I’ve dealt with bhyve VMs and jails during my time as a TrueNAS Core user, and they work well on Scale’s predecessor. But I’ve also spent weekends trying to install drivers and configure packages on FreeBSD systems, only to run into a cascading chain of errors. But this experimental tool, Sylve, had one little thing that caught my eye. Not its bhyve VMs, jail environments, or ZFS pools; Sylve had an interface uncannily similar to Proxmox. A juxtaposition of a simple, lightweight UI that’s built on top of something as complex as FreeBSD.
Now that I’ve used it for a couple of days, I have to admit that Sylve has really impressed me. If that sounds interesting, here’s a detailed log of my misadventures with this FreeBSD alternative to Proxmox.

                        The installation script made configuring Sylve a cakewalk
               

            I took the easy road, and that made all the difference
    
Most folks, including yours truly, install Proxmox the old-fashioned way – flash the ISO on a USB drive, run the installation wizard, and voilà. But if you’ve ventured into its documentation, you may be aware that it’s essentially a bunch of tools built on top of Debian. To that end, you can even deploy it on a pre-existing Debian installation. Sylve works the same way, except it doesn’t have a dedicated ISO (yet) and needs to be set up on a machine that’s already running FreeBSD.
✕
 Remove Ads
I won’t bore you with the details of how I installed FreeBSD, as it’s as simple as setting up any ol’ OS. Just so we’re clear, I used my outdated Ryzen 5 1600 PC for this project, partly because I’ve had decent luck setting up wacky operating systems on a system this old, and also because it’s currently a part of my secondary Proxmox cluster (yeah, I swapped the SSD bearing my PVE setup with a spare one). I had some minor issues getting an IPv4 address for the system, but I resolved this problem by switching from DHCP to a static IP. Once I’d finally logged into FreeBSD, I updated the system and configured SSH access (for troubleshooting), before looking into Sylve’s documentation.

After trying to wrap my head around all the complex-sounding steps, I noticed an automated installation script in one of the newer releases. Since I had nothing to lose, I ran 
fetch -o- 
https://sh.sylve.io
 | sh
 in the FreeBSD shell (though apparently the 
pkg install sylve
 command can accomplish this as well), fully expecting it to throw a bunch of errors. But to my surprise, the script simply requested my approval to install extra packages before working its magic. Within a few minutes, I had a Sylve instance up and running. Once I’d entered the IP address of the PC followed by port
 8181
, I typed the credentials generated by the installer to finally access Sylve’s UI.
✕
 Remove Ads

            Sadly, I had to set up the network stack manually
    
I wouldn’t say that Sylve’s interface is a 1:1 clone of Proxmox, but there are a couple of similarities between the two. Both have a Datacenter section, which houses the options for governing a cluster of hosts, as well as detailed settings for each node. The resource monitoring dashboard, notification window, and VM/LXC (jail in Sylve’s case) wizards have the same layout on both platforms. There’s even a dedicated Terminal tab in Sylve, where I can run commands, like the Shell section in Proxmox.
That said, Sylve’s settings are somewhat barebones when compared to Proxmox. Although it’s possible to modify the ZFS drive, it hides the storage pool setting within each node. The network section is more elaborate, sure, but it doesn’t have the hardcore SDN stack you’ll find on Proxmox.
While the automated installer took care of everything else, I had to set up the network objects and switches by myself. It’s not a dealbreaker by any means, and Sylve’s documentation highlights the necessary settings, so I had no issues creating fully-functional bridges for my virtual guests.

                        I had no trouble deploying Bhyve VMs via Sylve's Proxmox-like UI
               

            And the same holds true for FreeBSD and Linux jails
    
✕
 Remove Ads
Sylve doesn’t disappoint on the virtualization front, either. It relies on bhyve, a type-1 hypervisor on FreeBSD, to run virtual machines, with the VNC viewer acting as the main console for accessing them. Rather than splitting container templates and VM images in separate tabs, the Downloader section under the Utility submenu lets me add ISOs and 
tar.xz
 files to Sylve.
The VM deployment wizard is just as easy to use – a far cry from the time I tried managing bhyve using terminal commands. All I had to do was give it a name, specify the ISO file, choose the storage pool (and disk allocation), and grant it some resources. The actual number of tabs and options was fewer on Sylve’s virtual guest setup wizards, though I appreciate how it includes the VNC resolution within these settings. Soon, I had my 
Pop_OS!
, Garuda, and Void Linux virtual machines up and running, and since I’d allocated enough resources to them, their performance was nothing to scoff at, either.
I also wanted to experiment with jails, so I started looking at the Sylve documentation. Unlike the Proxmox VE Helper-Scripts or even TurnKey templates on its rival, Sylve requires some manual work when deploying jails. Well, I won’t say that jails are the same as containers, as they’re meant to be secure environments for isolated services rather than tools for packaging apps. But they’re the other virtual guests available on Sylve besides VMs, and it’s possible to configure both Linux and FreeBSD flavors. They’re not that complicated to deploy, either, as I simply downloaded the templates from the official documentation, extracted them, and followed the jail creation wizard, which includes interesting options like FStab additions, DevFS rules, and lifecycle hooks. Besides a FreeBSD jail, I also deployed one based on Rocky Linux, and even though Linux jails are supposed to be experimental, it was able to run typical CLI tools without throwing compatibility errors.

                        Sylve includes a handful of QoL services, too
               

            Despite its FreeBSD origins, Sylve is pretty darn useful for home labbers
    
The massive collection of quality-of-life services on Proxmox is a major factor behind its popularity, and surprisingly, Sylve manages to mimic a handful of these features. I can create snapshots for each virtual machine and jail on my Sylve server, and there are even provisions for creating offsite backups on remote nodes. Likewise, Sylve supports cloud-init templates, making it great for spinning up disposable virtual guests for DevOps and automation projects.

It even hides QEMU Guest Agent, Wake-on-LAN, and serial console emulation under the virtual machine settings. Technically, Sylve can emulate TPM implementation, but my attempts at getting Windows 11 VMs were met with the bhyve logo freezing at startup. Sylve includes a file browser baked into its web UI and a dedicated PCI passthrough page for letting virtual guests access extra devices – features that I’ve been desperately waiting for on Proxmox. Well, I couldn’t test PCI passthrough with anything except a USB expansion adapter, as my NICs and old GPUs don’t mesh well with FreeBSD. But still, kudos to the devs for adding some extra features for hardcore tinkerers.
✕

            Sylve can't replace Proxmox yet, but it exceeded all my expectations
    
The verdict? While I wouldn’t go about swapping every Proxmox instance with Sylve, I’ll admit that I’m really impressed by this FreeBSD platform. I’ve dealt with my fair share of cumbersome virtualization tools, and Sylve is not one of them by a long shot. That alone is worth praising, but what I absolutely adore about Sylve is that it works extremely well even when it’s built on top of FreeBSD. Sure, it may not include the sheer number of first-party apps, QoL features, and community repos like Proxmox, but its simple UI, robust virtualization tools, and essential VM/jail management utilities make it a terrific option for tinkerers looking to dabble with FreeBSD setups.
My only caveat is that I wasn’t able to get nested virtualization or 
GPU passthrough
 working on bhyve. Otherwise, I would’ve tried deploying something as wacky as a Proxmox instance or a gaming VM inside Sylve, just for giggles.

Hi! Author of Sylve here, thanks a lot for trying it out, this was a really fun read and super helpful for us.
> A juxtaposition of a simple, lightweight UI that’s built on top of something as complex as FreeBSD.
I’d actually argue FreeBSD is simpler than Linux! The init system, package management, and overall userland are much more consistent with far less fragmentation.
> Sylve works the same way, except it doesn’t have a dedicated ISO (yet)
This was intentional since most of what we need, especially ZFS, is already in base, so we didn’t see much value in maintaining a custom ISO.
> Sylve’s settings are somewhat barebones compared to Proxmox
That’s fair and definitely something we’re working on. FreeBSD networking is incredibly powerful and we’ve only scratched the surface so far. Advanced setups like bridges and LAGG are already possible for now (you can create them from CLI and import them as Manual bridges into Sylve)
> jails require some manual work
We’ve already improved this in master and it’ll be in the next release. We’re also working on Podman support to make app deployment much easier (app templates and container images)
> Windows 11 VM freezing
This is a known issue and already fixed upstream! Should be resolved in FreeBSD 15.1. We actually recommend using FreeBSD->=15-STABLE to our users as it'll get fixes/features much faster than release.
> GPU passthrough / nested virtualization
Nested virtualization is still a way off, but GPU passthrough works on our end so I’d love to help debug your setup. Feel free to reach out on chat(dot)sylve(dot)io!
Also worth mentioning, one of Sylve’s strengths is how resource efficient it is. The backend is written in Go, the UI with Svelte, so the system stays very lightweight. We’ve seen setups running with as little as around 384 MB of RAM even with ZFS, which makes it great for low-end hardware.
P.S. Had to trim my responses down a bit due to comment length, happy to answer any questions you may have!

I like proxmox but I love FreeBSD. I've been managing my jail instances using the extremely useful and somewhat underrated CBSD for the past few years. I tried vibe coding a proxmox clone in perl using the new object syntax. I got a decent prototype but I'm pretty sure the AI was just architecting my project based on the Sylve project. I made the mistake of wrapping libzfs using FFI::Platypus rather than just shelling out to zfs binaries because I do everything to hard way in order to never finish a project. It was mostly a learning experience since I knew my project would never be widely adopted in the FreeBSD or Perl communities. Anyhoo...
I switched to Sylve. It's a dream. I can't emphasize enough how great this project is and how grateful I am that it exists. I'm really excited for the future but the project does everything I was using CBSD for and more.  I think Linux is great for virtualizing on FreeBSD but I would never virtualize FreeBSD on Linux. Native ZFS solves so many fundamental systems management problems. That is really the strength of FreeBSD.  Sylve is the icing on the cake. I'm so excited that you have support from the FreeBSD Foundation. Good job!
