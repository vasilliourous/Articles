Cloudflare Workers is a serverless platform that lets you run JavaScript at the edge, handling requests close to users without managing servers. It powers fast APIs, routing logic, and lightweight apps with minimal setup. What’s less obvious is that the runtime behind it, workerd, is open source. That means you can replicate a similar setup on your own machine or server without relying on Cloudflare’s infrastructure. I tried exactly that: self-hosting a Workers-like environment using workerd, and it turned out to be far simpler than expected, both in terms of setup and actually getting something useful running locally.

                        Getting Workerd up and running
               

            It doesn't take a lot of time
    
Getting workerd running locally is surprisingly straightforward once you skip the unnecessary detours. The simplest path is to install 
Node.js
 using the official .msi installer from 
nodejs.org
. Don’t bother with Docker or package managers here; they add friction without helping. Once that’s done, open Command Prompt instead of PowerShell, since it tends to behave more predictably with Node and npx.
From there, create a working folder anywhere you like and move into it. Inside this folder, you’ll need just two files. The first is your actual worker script. Create a file called 
hello.js
 and drop in the following:
addEventListener
(
"fetch"
,
event
 =>
 {
event.
respondWith
(
newResponse
(
"Hello from workerd 🚀"
));
});

This is your entire application. Every request that hits your server will be handled here. The second file is where workerd gets interesting. It doesn’t use a typical config format but instead relies on Cap’n Proto. Create a file called 
config.capnp
 and add this:
using Workerd = import "/workerd/workerd.capnp";

const config :Workerd.Config = (
  services = [
    (name = "main", worker = .mainWorker),
  ],

  sockets = [
    (
      name = "http",
      address = "*:8080",
      http = (),
      service = "main"
    ),
  ]
);

const mainWorker :Workerd.Worker = (
  serviceWorkerScript = embed "hello.js",
  compatibilityDate = "2023-02-28",
);

This file wires everything together. It defines a service, binds it to a socket on port 8080, and tells workerd to execute your 
hello.js
 file when requests come in.
Once both files are in place, you don’t install workerd globally — you just run it directly using:
npx workerd serve 
config.capnp

The first run will download the runtime automatically. After that, it spins up a local server. Open your browser and head to http://localhost:8080, and you should see your response immediately.
It's worth noting how little infrastructure is actually needed. There’s no framework, build step, and complex routing unless you choose to add it. You’re effectively running the same runtime that powers Cloudflare Workers, but entirely on your own machine.

                        Workerd is Cloudflare Workers without Cloudflare
               

            It runs the same execution model
    
 Once you have workerd running, you'll notice that at its core, you’re working with the same execution model as Cloudflare. It’s also an event-driven JavaScript that responds to HTTP requests using standard Web APIs like fetch, Request, and Response. That means anything you’d typically build as a lightweight backend can live here, whether it’s APIs, authentication layers, redirects, or even small proxy services.
One of the more interesting capabilities is how workerd handles service composition. It lets you define multiple “nanoservices” in the same config and let them call each other internally instead of thinking in terms of separate microservices deployed across different machines. These calls don’t go over the network, so they’re extremely fast, almost like function calls. This is a very different model from traditional backend setups and even most serverless platforms, where services are isolated and communicate over HTTP.

You also get fine-grained control over routing and request handling. Because you define sockets and services explicitly in the Cap’n Proto config, you can decide exactly how traffic flows through your system. This makes it possible to use workerd as a programmable HTTP proxy, intercepting and modifying requests before they reach another service. That alone opens up use cases like custom caching layers, API gateways, or middleware-style processing without needing a full reverse proxy stack.

Compared to Cloudflare Workers, the biggest difference is everything around the runtime. With Cloudflare, you get global edge deployment, automatic scaling, built-in KV storage, and integrations like Durable Objects out of the box. With workerd, you only get the runtime itself.

            Workerd hands you the control
    
Workerd comes with some trade-offs, but it also hands you control. You’re not locked into a vendor, and you can run the same Workers-style code anywhere, from your laptop to a VPS cluster. If you’re willing to handle deployment and infrastructure, workerd gives you a surprisingly powerful foundation to build your own edge-like platform on your own terms.
